# 한눈 파악(glanceability) 설계 제안 (Phase 21)

사용자 시그널(2026-05-29, 두번째 핵심 원칙): "CLI claude 세션은 대화형·선형이라 길어지면
한눈에 파악하기 어렵다. Cortex 가 현재 무슨 일이 돌고 있고 프로젝트가 어떤 상태인지 한눈에
보여줘야 한다. 신중한 검토·좋은 아이디어가 필요하고 추측 구현은 금지."

→ 구현 전 방향 결정용 제안서. 추측 구현 대신 옵션을 제시하니 골라주시면 해당 안으로 진행한다.

## 진단 — "한눈에 안 보이는" 두 축

1. **세션 축**: claude CLI 위임 세션은 긴 대화 로그. 지금 어느 단계인지(탐색/구현/검증), 무슨
   결정을 했는지, 산출물(커밋·PR)이 뭔지 스크롤 없이 파악 불가.
2. **프로젝트 축**: 여러 레포의 "지금 상태"(진행 중 자동화, 대기 중 검토, 막힌 것)가 대시보드
   곳곳에 흩어져 있어 한 화면 종합 뷰가 약하다.

## 이미 있는 자산 (재활용)

- 대시보드 stat 5종(검토 대기·자동/수동 머지·진행 중·평균 신뢰), 최근 머지, 프로젝트 진척 카드.
- in-flight 자동화 배지(A1, `automation-state`), 알림(SSE + 토스트), 미확인 머지 배지(Phase 20).
- agent_runs(위임 상태), work-state.md(세션 연속성), roadmap 진척.

부족한 건 "데이터"가 아니라 **한 화면 종합 + 세션 단계 압축**.

## 제안 (옵션 — 골라주세요)

### G1. 라이브 상태 스트립 / "지금" 보드 ★ ROI 높음, 저위험

대시보드 상단(또는 사이드바 상시)에 **지금 돌아가는 것 한 줄 종합**: 진행 중 위임 N · 자동화
in-flight M(충돌해결/테스트수정/리뷰반영) · 검토 대기 K · 미확인 머지 U. 클릭 시 해당 목록.
- 데이터 전부 존재(agent_runs·automation-state·inbox·unreadMergedCount). 집계 쿼리 1개 + 얇은 UI.
- **추측 없음**: 기존 카운트의 재배치/종합일 뿐. 저위험·고가치. **1순위 추천.**

### G2. 위임 세션 단계 타임라인 (세션 축)

위임 세션을 "단계(탐색→구현→검증→PR)·핵심 결정·산출물"로 압축 표시. 긴 로그 대신 타임라인.
- **근거 데이터 필요**: 현재 agent_run 은 상태(running/done)만 안다. 단계/결정 추출은
  claude CLI `stream-json`(Phase 13.6 R6) 이벤트 파싱이 선행되어야 정확. 그 전엔 휴리스틱
  (커밋/PR 생성 시점 + 세션 출력 마커)으로 근사만 가능.
- **중간 노력·런타임 검증 필요**. stream-json 도입(R6)과 함께 단계적으로.

### G3. 비선형 요청 quick-capture (메타 원칙)

사용자가 흩어진 요청을 즉시 던져 넣는 곳. 어디서나(전역 단축키/헤더) 한 줄 캡처 → "캡처함"
목록 → 분류(이슈 승격 / 로드맵 항목 / TODO / 폐기). "요청이 누락되는 패턴 자체"를 Cortex 가 흡수.
- 데이터: 기존 todos/notes 재활용 가능(전용 `kind: 'capture'` 또는 분류 전 상태). UI: 전역 quick-add.
- **저~중 노력**. 사용자 워크플로(비선형 인터럽트)에 직접 대응 — 가치 높음.

### G4. 프로젝트 상태 한 줄 요약 (프로젝트 축)

각 프로젝트 카드에 "상태 한 줄"(예: "검토 2 · 진행 1 · 막힘 0 · 진척 62%")을 표준화. 대시보드
프로젝트 위젯 강화. G1 의 프로젝트 단위 버전.

## 권고 진행안

1. **G1(라이브 상태 스트립)** — 저위험·고가치, 기존 데이터 종합. 먼저.
2. **G3(quick-capture)** — 사용자 메타 원칙 직접 대응. 다음.
3. **G2(세션 타임라인)** — stream-json(R6) 런타임 검증 후. 데이터 기반 신중 진행.
4. G4 는 G1 과 묶어도 됨.

## 사용자 결정 필요

- 어느 안부터? (추천: G1 → G3) 또는 다른 형태(예: 전용 "상황판" 페이지 vs 대시보드 통합).
- 상태 스트립 위치: 대시보드 상단 vs 전역 사이드바 상시 vs 헤더.

결정되면 G1 은 자율 구현 가능(집계 lib + 테스트 + 읽기 전용 UI). G2 는 R6 선행.
