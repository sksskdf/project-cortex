# Design — Project Cortex

> **직관적이면서 단순할 것.** 이 외에는 모두 부차적입니다.

프로토타입(`prototype/*.html`)은 _참고_입니다. 화려한 디테일이 있어도 따라가지 않아도 됩니다. **단순함을 해치는 디테일이라면 빼는 게 정답입니다.**

---

## 1. 디자인 원칙 (5개로 줄임)

### 1.1 한 화면 = 한 가지 일

화면이 하는 일을 한 문장으로 못 쓰면 화면을 둘로 쪼개세요.
- 인박스 = "지금 봐야 할 PR을 우선순위 순으로 본다."
- PR 리뷰 = "이 PR을 머지할지 결정한다."
- 클러스터 = "묶인 PR 5개를 한 번에 처리한다."

보조 기능(통계·차트·바로가기)은 디폴트로 보이지 않게 하거나, 별도 화면으로 옮깁니다.

### 1.2 디폴트는 안 보임 (Progressive disclosure)

- 자신 있는 hunk는 **접혀 있음.** 펼치고 싶을 때만 펼쳐서 본다.
- 모든 메타 정보를 카드에 우겨넣지 않는다. 필요한 3개만.
- "더 보기" 링크는 정말 더 볼 게 있을 때만.

### 1.3 색 = 의미

| 색 | 뜻 | 토큰 |
|---|---|---|
| 파랑 | 기본 액션, 활성 상태 | `--ds-color-secondary-01` |
| 초록 | 통과, 자동 승인 가능 | `--ds-color-state-success-*` |
| 노랑 | 주의, 사람 확인 권장 | `--ds-color-state-warning-*` |
| 빨강 | 위험, 사람 승인 필수 | `--ds-color-state-error-*` |
| 보라 | 정보성 태그 | `--ds-color-soft-purple` |
| 회색 | 보조 텍스트, 비활성 | `--ds-color-text-02`, `--ds-color-soft-gray-*` |

장식용 색 금지. 같은 의미는 항상 같은 색.

### 1.4 신뢰 점수 매핑 (고정)

| 점수 | 색 | 의미 |
|---|---|---|
| 90+ | 초록 | 자동 머지 가능 영역 |
| 70–89 | 파랑 | 빠른 확인 권장 |
| 50–69 | 노랑 | 사람 검토 필수 |
| <50 | 빨강 | 위험 — 우선 처리 |

**이 매핑은 변경하지 않습니다.** 모든 화면에서 같은 매핑.

### 1.5 클릭 한 번이 디폴트

- 인박스 → PR 머지까지 클릭 3회 이내.
- "어디 클릭하지?" 묻게 만들면 안 됨. 액션은 항상 명백한 위치 (헤더 우측, 카드 우측, sticky bottom).
- 사람이 가장 자주 하는 결정은 **빠른 액션 버튼** (인박스 행의 ✓ 승인 같은 것)으로 노출.

## 2. 레이아웃 (단순 룰 세 개)

1. **2분할이 디폴트.** 좌측 240px Sidebar + 우측 메인. 더 복잡한 분할은 메인 안에서 처리.
2. **메인 영역 안에 보조 레일(rail)을 둘 때는 240–280px.** 그 이상이면 화면 두 개로 쪼개세요.
3. **카드 vs 행** — 항목이 시각적으로 동등하면 행(list). 위계가 있으면 카드(card).

데스크톱 1280px+ 우선. 모바일 반응형은 별도 세션.

---

## 3. 타이포 (외울 것 4개)

- 페이지 타이틀: 28px / Bold / tight letter-spacing
- 섹션 타이틀: 18px / Bold
- 본문·라벨: 14px / Regular
- 보조: 12–13px / Regular / `text-02` 색

폰트 패밀리는 자동 — `var(--ds-typography-font-family-text)`. 숫자만 등장하면 `--ds-typography-font-family-number`(Spoqa)로 감쌉니다.

---

## 4. 인터랙션 (그게 다)

- **Hover**: 색 한 단계, 또는 살짝 lift (`translateY(-1px)`).
- **Press**: `scale(0.98)`. 모든 버튼.
- **Focus**: `box-shadow: 0 0 0 3px var(--ds-color-secondary-03)`. outline 없음.
- **Transition**: 0.15s `ease`. 그 이상 길게 가지 않음.

애니메이션 라이브러리 도입 안 함. 화려한 entry animation 안 함.

---

## 5. 카피 룰 (강함)

- 한국어 존댓말 (`~해주세요`, `~합니다`).
- 라벨·버튼: 12자 이내. 점 없음.
- 본문·헬퍼·에러: 한 문장. 점 있음. 책임 회피 표현 금지 ("오류가 발생했습니다" X → "이메일 형식을 확인해주세요." O).
- 명령형(`~하라`), 이모지, all-caps, 반말 — 0건.

모든 카피는 `src/copy/ko.ts`에 키로 둡니다. 인라인 한글 텍스트 금지.

---

## 6. UI 컴포넌트 (이미 있는 것을 쓰세요)

새 컴포넌트를 만들기 전에 다음을 확인:

| 필요한 것 | 이미 있는 곳 |
|---|---|
| 버튼 | `lib.css`의 `.ds-btn .ds-btn--md .ds-btn--filled-blue` 등 |
| 카드 컨테이너 | `.ds-card` (radius 20px 시그니처) |
| 태그 (상태/위험) | `.ds-tag .ds-tag--{red,yellow,cyan,sky-blue,purple,gray}` |
| 칩 (필터) | `.ds-chip` 변형들 |
| 입력 | `.ds-input .ds-input--md` |
| 세그먼트 (탭형 토글) | `.ds-segment` |
| 뱃지 (카운트) | `.ds-badge` |
| 신뢰 점수 게이지 | `<Gauge>` 컴포넌트 (만들고 나면) |
| Sidebar | `<Sidebar>` 컴포넌트 |
| Author 칩 (에이전트/사람) | `<AuthorChip>` 컴포넌트 |

**이미 있는 것을 prop만 바꿔 쓰세요.** 비슷한 새 컴포넌트를 만들면 일관성과 토큰 모두 깨집니다.

## 7. 의사결정 가이드 (헷갈릴 때)

| 헷갈릴 때 | 따르는 룰 |
|---|---|
| 이걸 추가할까 말까? | **빼는 쪽이 디폴트.** 사용자가 명시적으로 원할 때만 추가. |
| 더 예쁘게 만들까? | **단순한 쪽이 디폴트.** "이미 충분히 명확한가?" 물어 답이 예스면 멈춤. |
| 정보를 더 노출할까? | **숨기는 쪽이 디폴트.** 디폴트로 안 보이면 1단계 펼침에 둠. |
| 클릭 1회 vs 2회? | **1회.** 항상 1회를 추구. |
| 색을 추가할까? | **No.** 5색(파랑/초록/노랑/빨강/회색) 안에서 해결. |
| 새 아이콘 필요? | 디자인 시스템 아이콘에서 찾기. 없으면 가장 가까운 거 재사용. |
| 텍스트가 길어요 | **짧게 만들기.** 12자에 안 맞으면 카피 자체가 잘못된 것. |

## 8. 안티패턴

다음은 모두 **단순함을 깨는** 사례입니다 — 발견하면 제거:

- 한 화면에 통계 위젯 + 차트 + 리스트 + 활동 피드를 다 우겨넣기 (대시보드 함정)
- "고급" 토글로 더 많은 옵션을 노출
- 사이드바 내 깊이 3 이상의 트리
- 같은 동작에 두 가지 버튼 (예: 헤더에 "저장" + 푸터에도 "저장")
- 색이 의미 없이 다양 (브랜드 색 + 그라데이션 + 무지개 차트)
- 모달 안의 모달
- Tooltip이 핵심 정보를 담음 (호버해야 보이는 정보는 핵심 아님)
- "곧 출시" / "준비 중" 같은 빈 placeholder를 화면에 노출
