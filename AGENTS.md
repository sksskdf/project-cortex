# AGENTS.md — Project Cortex 작업 규칙

> 이 파일은 AI 에이전트(Claude Code, Devin, Codex 등)가 Cortex 코드베이스에서 작업할 때 따르는 룰입니다.
> **다른 어떤 문서보다 먼저 읽으세요.** 한 번 읽으면 다음 작업부터 추가 컨텍스트를 거의 안 읽어도 됩니다.

## 0. 첫 작업 시 읽는 것 (전부)

1. **`README.md`** — 제품 컨셉, 4대 원칙. 1분.
2. **`AGENTS.md`** (이 파일) — 작업 룰. 3분.

이 둘로 90%의 작업이 가능합니다. 모자라면 그때 `docs/` 안의 해당 파일만 추가로 봅니다.

| 모자랄 때 추가로 보는 것 | 언제 |
|---|---|
| `docs/ARCHITECTURE.md` | 새 모듈을 만들거나 폴더 구조에 손댈 때 |
| `docs/DOMAIN.md` | DB 스키마·도메인 타입·트라이아지/클러스터링 룰을 다룰 때 |
| `docs/DESIGN.md` | 새 UI 화면·컴포넌트를 만들 때 |
| `docs/CONVENTIONS.md` | 네이밍·임포트 순서 등이 헷갈릴 때 |
| `docs/ROADMAP.md` | 지금 Phase가 어디고 다음 산출물이 무엇인지 확인할 때 |

전체 컨텍스트를 한 번에 다 읽지 마세요. **모르면 그때 찾는다**가 디폴트입니다.

## 1. 토큰 경제 (Token Economy) — 핵심 룰

이 프로젝트는 AI가 코드를 짜는 워크플로우를 다루는 도구입니다.
**도구가 다루는 가치(=토큰 효율)는 도구 자체가 먼저 실천합니다.**

### 1.1 작업 시작 전

- **작업 1건 = 파일 1–3개 변경**이 디폴트. 5개 이상 손대게 되면 멈추고 PR을 쪼개세요.
- 무관한 파일을 "겸사겸사" 정리하지 마세요. 클러스터링이 깨지고 사람 리뷰가 늘어납니다.
- 이미 있는 것부터 검색합니다. 새 컴포넌트·헬퍼·카피를 만들기 전에 `src/components/`, `src/lib/`, `src/copy/`를 grep하세요.

### 1.2 코드 작성 중

- **카피는 인라인 금지.** 모든 한국어 텍스트는 `src/copy/ko.ts`에 키로 두고 `t('inbox.empty')`로 참조합니다. 같은 문구를 두 번 쓰지 않습니다.
- **색·간격·radius는 토큰만.** `var(--ds-color-secondary-01)` 등을 직접 씁니다. 새 색을 만들지 마세요. (정확한 토큰명은 `prototype/design-system/colors_and_type.css`에 있습니다 — grep 한 번이면 찾습니다.)
- **레이아웃은 `<AppShell>` 사용.** Sidebar 마크업을 다시 작성하면 안 됩니다 — 한 번 정의된 레이아웃을 import해 씁니다.
- **공유 시각 요소(`<Gauge>`, `<AuthorChip>`, `<PRRow>` 등)는 `src/components/`에서 가져옵니다.** 비슷한 게 없으면 만들고, **있으면 props로 확장**합니다.
- **외부 시스템 대체 데이터(git diff·LLM 출력·운영 메트릭 등)는 `src/fixtures/`에서 import.** 컴포넌트 안에 데이터 박지 마세요. 화면 수정과 데이터 수정이 분리되면 토큰이 절약됩니다.

### 1.3 응답 작성 중

- 변경 결과를 설명할 때 **변경한 파일과 핵심 라인만** 언급합니다. 코드를 다시 출력하지 마세요 (사용자가 diff로 봅니다).
- "다음에 할 수 있는 것" 같은 제안은 한 줄로. 불필요한 트레이드오프 비교를 늘어놓지 마세요.

### 1.4 자기 점검

코드를 쓰기 전에 스스로에게 묻습니다:
- "이 마크업이 이미 어딘가에 있지 않나?" → grep
- "이 카피가 `ko.ts`에 이미 있지 않나?" → grep
- "이 색·간격이 토큰으로 있지 않나?" → 거의 항상 있음
- "내가 지금 손대는 파일이 이 작업과 정말 관련 있나?" → 아니면 멈춤

## 3. 스택 (한 줄)

- **Next.js (App Router) + TypeScript + SQLite(Drizzle) + Urock 디자인 시스템 CSS 그대로.**
- GitHub 연동은 `@octokit/rest`를 `src/lib/github.ts` 어댑터로 감싸서 사용하고, 테스트는 Vitest(`npm run test`)를 기본으로 사용합니다.
- Tailwind 안 씁니다. 디자인 시스템 토큰을 CSS Variable로 직접 사용합니다.
- 폼 검증은 `zod`, 서버 호출은 React **Server Actions** 우선 (API route는 webhook 같은 경우만).

자세한 구조는 `docs/ARCHITECTURE.md`.

---

## 4. 작업 룰 (체크리스트)

새 화면·기능을 만들 때 이 순서를 지킵니다:

1. **카피 먼저** — `src/copy/ko.ts`에 키를 추가합니다.
2. **타입 먼저** — 도메인 객체 변경이 있으면 `src/lib/types.ts`에 먼저 반영합니다.
3. **데이터 먼저** — DB 시드(`src/db/seed.ts`)나 외부 시스템 대체 fixture(`src/fixtures/`)에 데이터를 만들고, UI는 `src/lib/*` 함수로 받습니다.
4. **AppShell 안에서 작업** — 새 라우트는 `src/app/*/page.tsx`로 추가합니다.
5. **공유 컴포넌트 재사용 우선** — 없으면 만들고, 만들면 다른 화면에서도 쓸 수 있게 props로 분리합니다.
6. **테스트는 가벼운 것만** — `*.test.ts` 한 파일에 5–10개 케이스, 비즈니스 로직만. UI 스냅샷은 안 합니다.

## 5. 절대 하지 말 것

- ❌ 인라인 한국어 문자열 (`<h1>인박스</h1>` X → `<h1>{t('inbox.title')}</h1>` O)
- ❌ 인라인 hex 색 (`color: #628cf5` X → `color: var(--ds-color-secondary-01)` O)
- ❌ Sidebar/Header 마크업을 새 페이지에 다시 작성
- ❌ "스타일 정리"라는 이름의 무관한 변경 끼워 넣기
- ❌ 새 라이브러리 추가 시 사람 확인 없이 진행 (특히 UI 라이브러리·런타임 변경)
- ❌ 이모지, `alert()`, `confirm()`, `prompt()`
- ❌ Cortex 자체의 자동 머지 정책을 우회하기 위한 trivial 변경 (정직하게 만드세요)

## 6. 도구 호출 가이드 (Cortex 자체 기능)

코드를 짜는 중 도움이 필요하면:

- **로컬 grep**으로 충분한 것은 grep으로. AI 호출이 가장 비쌉니다.
- **사람 확인이 필요해 보이면 멈추고 묻습니다.** 추측으로 진행하지 마세요 — 잘못된 100줄보다 1번의 질문이 쌉니다.
- **빌드·테스트는 한 번에 묶어서.** 매 변경마다 `npm test` 돌리지 마세요.

## 7. 카피 톤 (강한 룰)

- 한국어 존댓말 (`~해주세요`, `~합니다`, `~인가요?`)
- 명령형 금지 (`~하라` X)
- 12자 이내 라벨이 기본. 안 되면 줄이세요.
- 마침표: 본문·헬퍼·에러는 끝에 점, 라벨·버튼은 점 없음.
- 이모지·all-caps·반말 0건.

전체 카피 사전: `src/copy/ko.ts` (작업 시작하면 자동 생성).

## 8. Git 워크플로우 (강한 룰)

- 모든 작업은 별도 브랜치에서. **master 직접 푸시 금지.**
- 브랜치명: `feat/`, `fix/`, `chore/`, `docs/`, `refactor/` 접두사 + 짧은 이름.
- master로 **squash merge.** 한 PR = 한 변경 (§1.1과 동일 원칙).
- PR을 올릴 때 `.github/PULL_REQUEST_TEMPLATE.md`의 체크리스트를 채웁니다.

자세한 룰은 `docs/CONVENTIONS.md` §10·§13.
